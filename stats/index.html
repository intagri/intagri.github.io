統計不正問題とITの日常

Publish date: Feb 6, 2019
Tags: data 統計 IT

厚生労働省が所管する毎月労働統計に不正があったということで、議論になっています。単なるミスなのか、意図あるミスなのかという議論に加えて、集計システム開発にコボル（COBOL）というコンピューター言語を使っていた（なんら問題ないはずです）ということがさらに議論を混迷させています。

どんな議論か

世界銀行のDoing Business Reportのランキングに不正があったのではないという話を以前のブログで取り上げましたが、公的機関が出す統計や数字については一度信頼を失うと回復するのが大変です。さまざまな利用者が、それらの統計に基づいて判断、行動するわけですので、間違いましたので訂正しますではすみません。

厚労省は原因究明と再発防止策策定のために特別監査委員会を立ち上げ、その報告者が1月22日に公表されました。その中で、システム改修すべきところをしなかった経緯と原因について触れています。システムがCOBOLで開発され、当時COBOLを扱える職員が2人しかいなかったとの説明がありますが、原因はシステム改修体制が「事務処理に誤りが生じやすく、発生した事務処理の誤りが長年にわたり発見されにくい体制となっていたこと」にあるとしています。

COBOLでシステムを開発したことに原因があるわけでもなく、COBOLを理解している職員が2人しかいなかったことが原因となったわけでもありません。ところが、システム改修のミスをCOBOLと関係つけるような記事が出てきて盛り上がっているようです。

COBOL（Common Business Oriented Language）は今年で生誕60年を迎える事務処理専用のコンピューター言語です。世界で多分3番目くらいに古く、いまだに現役の言語です。かつて大型電算機（メインフレーム）全盛の頃は世界中の民間、政府で使われていましたが、メインフレームからサーバーへ移行、代替する言語の出現などによりユーザーが減少、現在では金融、保険、公的機関など主にに巨額の金額計算を必要とされているところで使われています。シェアの減少に伴い開発、保守、運用の要員が減少し高齢化も進んでいることから、COBOLの将来を不安視する議論が継続しています。この辺りが、COBOLは死んだと言われ、何かあるとCOBOL不要論につながっているようです。

以前のブログでシステム部門の悲哀のような話を書きましたが、今回の統計システムをめぐる話もITの日常であり、COBOLの日常なのです。この日常について、 個人的な経験に触れておきたいと思います。

枯れた技術

15、6年前、IT部門の経験がないまま情報システム部門の責任者となりました。鯖おやじ丸出しなのですが、近く来るであろうシステム更新にあたって、引き続きCOBOLでいくのか、それともJavaを使うのかを担当者たちに聞いてみました。いろんな議論がありました。とくに記憶に残っているのは、枯れた技術でないといけないという信念のような意見でした。

COBOLか、Javaかというのは瑣末な議論で、枯れた技術かどうかというのが重要。我々はシステムの面倒を見ているように思われがちだが、金融勘定データーを守っていくのが仕事。データーが喪失したら、組織は終わり。データーを守る道具がシステムであり、仮に今後10年間稼働させるシステムを作るのであれば、堅牢で安定している技術を選ぶのが我々の仕事となる。そういう枯れた技術とみなせるのであれば、COBOLでもJavaでも構わない。ただし、現時点でJavaはまだ枯れた技術とは言えない。

大規模で10年以上は稼働させようというシステムにおいては、たとえばバージョン・アップするだけでも、大変な作業になります。どんな言語であれ、バージョン・アップによってシステムに不都合が生じる可能性は否定できないので、事前にバージョンアップによる悪影響がないかというテスト作業が必須です。うまくバージョン・アップできれば良いですが、最悪の場合、最新バージョンに下位互換性がなくなり、最新のバージョンでは以前のバージョンで作ったプログラムが動かなくなるというようなケースもありえます（最近の例では、python3とpython2）。システムの長期稼働を考えれば互換性重視という言語（枯れた技術）を選択するのは合理的なのです。

枯れた技術という概念が一般的かどうかはわかりませんが、同じような例えを見ることができます。

Haskellという言語の開発者の一人であるSimon Peyton-Jonesがその講演で、ある言語が不死の敷居（Threshold of Immutability）」を超えられるかという説明をしています。先ほど触れたJavaという言語も一時期、終わった、死んだと言われたことがあります。その頃（2008年）、IBMの技術者がJavaの将来性についてCOBOLを引き合いにしつつ書いた文章があります。COBOLのように死んだ言語という一文で「不死の敷居」を引用しつつ、JavaはCOBOLなどと同じようにこの先も使い続けられる述べています。以下、COBOLについて触れてあるところです。

ところでメインフレームと言えば、COBOL がそうした第 1 位の座を降りてから今や何十年にもなりますが、それでもなお COBOL は、無数の業界評論家達によって「死んだ」と断言されたにもかかわらず、小切手の現金化や預金からの振込み、クレジット・カードの支払い、そして主な金融ネットワークを実行するために使われ続けています。墓場の中で腐敗しているはずのものにとっては悪い話ではありません。私はこのことからマーク・トウェインを思い出します。彼は、彼の故郷の新聞に自分自身の死亡記事が掲載されたのを見せられた時、「皆さん、私の死亡記事は非常に誇張されています」と言ったそうです。

Java 言語は、そしてもっと広い意味での Java プラットフォームとその関連エコシステムは、C++ や C、COBOL その他の言語と同様、Simon Peyton-Jones が言うところの「不死のための敷居」をずっと以前にまたいでいます。そうした言語は、ほぼ永遠に存在し続けるツールです。なぜならそれらの言語は有用な目的に使われているか、あるいはそれらの言語のコードを再作成するためのコストが、それらの言語のシステムを「そのまま」使用して維持し続けるためのコストに比べて膨大になるからです。(ある特定の言語またはシステムが前者のカテゴリーに当てはまるのか後者のカテゴリーに当てはまるのかには大いに議論の余地がありますが、この記事ではその点はまったく重要ではありません。)

厚労省の統計システムがどれくらい複雑かつ大きいものかわかりませんが、仮にこの先、引き続きCOBOLを使って更新したとしても、なんら問題はないでしょう。

スタンプ・ラリー

日本の多くの組織では、稟議という文書を使って意思決定を行っています。担当者が文書を作り、上位の意思決定者に至るまでの複数の関係者全員のハンコをもらって決定が完了するというものです。今の若い人たちの中には、これをお父さんたちのスタンプ・ラリーと呼んでいます（広く広まっているかはわかりませんが）。

さて、このスタンプ・ラリーがコンピューター・システムとどのように関係があるのか、不思議に思われるかもしれません。組織におけるシステムは、売る、買う、貸す、借りるなどの取引によって生じるデーターを人の手ではなく、システムによって処理しようとするものです。ですから、スタンプ・ラリーによって決定された取引のデーターをどこかのタイミングでシステムに正確に入力してもらわなければなりません。

ところが、スタンプ・ラリーにかまけて、データー入力を忘れた結果、いわゆる事故が発生することがありました。このような事故を避けるため、システム設計担当者は、スタンプ・ラリーの事務フローを分析して、どのタイミングでシステムに入力させれば負荷も少なく、事故も防げるかと腐心してきました。

たとえば、ある取引のスタンプ・ラリーをするためには、システムに必要事項を入力して、その結果として出力されるスタンプ・ラリーの台紙（＝稟議書）を使うことで、システムとスタンプ・ラリーを開始段階で結びつけることができます。もちろん、ラリー完了後に意思決定済みという入力を忘れる可能性がありますので、すべてのスタンプ・ラリーを電子決済にするというのが近年の流れです。

どの言語でシステム開発するかなどの議論よりも、開発されたシステムを使って仕事がミスなく、円滑かつ効率的に行えるようになったか、どうかが重要です。

先ほどの枯れた技術の話をしていた頃、若手職員が異動となって情報システム部門にやってきました。営業企画で活躍していた職員だったので不思議に思い、人事に異動の理由を確認すると、近くマッキンゼーとかボスコンとかに転職するのでそれまであづかってくれということでした。彼が退職する前日、ここで働いてどうだったかと質問しました。

若手）ここで働いて非常に良かったと思っています。

私）なんで？

若手）誰が組織を動かしているのかが、わかりました。XXさんとZZさんが、本当の意味で組織を動かしているということがわかりました。

私）どういうこと？

若手）企画では、いろんなことを決めて、組織を動かしているような気分でした。しかし、ここに来て初めてそういう決め事が絵に描いた餅にならないように、誰かが具体的に事務フローに落として、システムに取り込んで決め事が本当に動くようにしているのがわかりました。もし、あの2人がいなければ、頓挫する、事故が多発する事態になるケースが出ていたと思います。

私）ほおー。

若手）組織の特徴として、企画力があるとか、リーダシップがあるとかよく言いますが、大前提としてそのような組織にはあの2人のような職員がいるのだと確信しました。ここに来るまで、情報システムはコンピューターだけ扱っているのだと思っていました。違いました。本当に間違った認識をしてました。

対応策はあるのか

1970年代、80年代、各国政府は行政サービスにコンピューター・システムを導入していきます。しかし、導入後、当初想定していた行政サービスが改善されていないのではないかという批判に直面します。1980年代、英国政府はこの批判に対応して、ITを使った行政サービス改善のための方策を模索し始めます。その集大成が1986年、ガイドラインとしてまとめられ、これは現在ITIL（Information Technology Infrastructure Library）という国際規格になっています。

日本政府も、おそらくITIL策定と同様の問題意識のもとガイドラインを策定しています。標準ガイドラインというもので、2014年頃に初版が出て、定期的に改定されています（年度内にも改訂版が出るようです）。

今回の統計不正問題について、特別監査委員会の報告を読むと、主として統計行政の観点から調査・分析され、統計行政の問題とITガバナンスの問題が混在しています。両者は別の問題であり、対処策も異なります。

統計とITという2つのガバナンスが相互に協力、牽制していくことが求められているはずですが、今回その辺りに空白が生じていたようです。統計部門のシステムであっても、政府CIO、府省CIOのガバナンスのもとで標準ガイドラインに沿ってシステム運用・保守作業を行なっているわけですから、担当部長・課長と担当者に問題がある（否定はしませんが）という結論ではなく、標準ガイドラインに沿って2014年以降、何ができていて、何ができていなかったのかを調査・分析しない問題の所在が明確とならず、再発防止策も策定しようがありません。

政府CIO、府省CIOがここで頑張らないと、各省庁、省内各部局独自のITガバナンス（もしあれば）にお任せする体制が継続することになりかねません。

3月27日付日経新聞の「崩壊厚労省（中）」という記事で、コボルが取り上げられていました。該当部分です。

使い手は二人だけ

統計不正は省内でも長年にわたり見過ごされてきた。毎月勤労統計のシステムは60年前に生まれたプログラム言語「コボル」で書かれている。90年代初めに稼働し、小幅な改修を積み重ねたが、不正が始まる直前の03年に省内でコボルを扱える職員はたった2人であった。

コボルは、IT（情報技術）業界で10年以上働く30代男性が「民間で使うエンジニアを見たことがない」と話す古い言語だ。厚労省でも使い手が1〜2人の状況が続き、係長以下がシステム改修の実務を担い、課長ら幹部は関与しないことが常態化した。調査対象を勝手に減らした04年、統計上必要な復元加工するシステムは施されなかった。手が回らなかったのか、そもそも計画がなかったのか。当時の担当者はすでに亡くなり、真相は不明だ。

この記事を読んでCOBOLが問題だと感じた人は多いのではないでしょうか。しかし、問題のシステムが開発された90年代初めころ、COBOLを使うことは当たり前でした。おそらく、日経新聞でも普通に使っていたと思います。その当時、社内にエンジニアを抱えるのが普通でしたが、その後作業を外注することでエンジニアを減らしていく傾向となり、これは日経新聞も厚労省も同じでしょう。たった2人でも外注できるのであれば、なんら問題ないのです。日経新聞本社がCOBOLについて特定の意見を持っているのどうかわかりませんが、日経グループ会社のコンピューター雑誌はCOBOLの関係者が作るCOBOL Consortiumとそのセミナーを長らく支援してきています。これも、ITの日常であり、COBOLの日常なのです。
